# Post-mortem: Google auth on an MCP server hosted on Zo

Date: 2026-03-07
Repo: `google_workspace_mcp`
Service: `gwmcp-auth`

## Executive summary

We spent most of the debugging time chasing what looked like a Google OAuth misconfiguration. The deeper issue was that there were actually **multiple independent failure modes** stacked on top of each other:

1. **Cross-process OAuth state loss**
2. **Ambient/inherited Zo environment variables colliding with app-specific Google OAuth config**
3. **Stale one-time auth URLs being reused**
4. **Confusion between “saved debug URL in a file” and “fresh live URL generated by the running process”**

The important lesson is that MCP auth on Zo is not just “configure Google correctly.” It is a three-layer system:

- **app code** decides how OAuth state and redirect URIs are generated
- **Zo service runtime** decides which environment variables the process actually sees
- **user/browser flow** decides whether a one-time URL is fresh or stale

When any of those three are even slightly wrong, the observed error can be misleading.

## Goal

Deploy the fork under this policy:

- calendar writes only to primary calendar
- empty attendee allowlist by default
- no invites / no guest invite permissions
- Gmail access limited to readonly
- Calendar access allowed for write operations on primary only

The policy fork itself was fine. The blocker was the OAuth path.

## What was true

### 1. The policy fork was not the problem
Targeted tests passed, and the permission-reduced startup mode behaved as intended.

### 2. `redirect_uri_mismatch` was real, but not the whole story
That error appeared after the cross-process state bug was patched. It was not imaginary. But it was masking a second issue: the app was likely picking up the wrong Google OAuth client from the environment.

### 3. Zo can provide ambient environment variables that your app did not explicitly set
Using generic names like:

- `GOOGLE_OAUTH_CLIENT_ID`
- `GOOGLE_OAUTH_CLIENT_SECRET`

is dangerous in Zo because those names may already exist in the runtime environment for other system purposes. If the app blindly prefers those names, it can silently authenticate against the wrong Google OAuth client.

### 4. One-time OAuth URLs are actually one-time
A saved auth URL in a markdown file is not a durable artifact. It is only valid as long as its state is still pending in the running state store.

## Root causes

## Root cause A: OAuth state was process-local, but callback handling was cross-process
Original behavior:

- one process generated the auth URL and stored `state` in memory
- a different process handled the callback
- callback process had no matching in-memory state
- result: `Invalid or expired OAuth state parameter`

Fix:

- persist OAuth state to disk in a shared JSON file

Lesson:

- if auth initiation and callback can touch different processes, **OAuth state cannot live only in process memory**

## Root cause B: generic Google env var names were too broad for Zo
Original behavior:

- app preferred `GOOGLE_OAUTH_*`
- Zo runtime appears capable of providing those names ambiently
- app could therefore load a different client than the one intended for this repo/service
- result: token exchange could fail with `redirect_uri_mismatch` even when the Google Cloud console screenshot looked correct

Fix:

- patch the repo to prefer workspace-specific names first:
  - `WORKSPACE_GOOGLE_OAUTH_CLIENT_ID`
  - `WORKSPACE_GOOGLE_OAUTH_CLIENT_SECRET`
- keep generic `GOOGLE_OAUTH_*` only as fallback

Lesson:

- on Zo, **do not rely on generic provider env var names** for an app you want isolated
- use app-scoped names first

## Root cause C: stale auth URLs were reused
Original behavior:

- an old `google-auth-url.md` existed in the repo
- it looked like “the auth URL,” but it was just an old saved string
- users can also retry old tabs or refresh callback pages
- once the state is consumed or expires, the link is dead
- result: repeated `Invalid or expired OAuth state parameter`

Fix:

- generate a fresh URL from the live process each time
- use it once, immediately
- do not rely on saved files as the source of truth

Lesson:

- **the source of truth is the live state store, not a markdown note**

## What finally worked conceptually

The correct pattern was:

1. patch the app to use app-specific OAuth env vars first
2. put the actual secret values into the Zo service environment
3. restart the hosted service
4. generate a fresh auth URL from the current code path
5. click it once immediately
6. complete consent without reusing old tabs or old URLs

## What made this confusing

### Same symptom, different causes
`Invalid or expired OAuth state parameter` can mean:

- wrong process generated vs handled callback
- stale URL
- state already consumed
- state file empty because no fresh auth flow was initiated

`redirect_uri_mismatch` can mean:

- actual redirect mismatch
- wrong client loaded from env
- correct redirect but wrong client-secret pair

So the visible Google/OAuth error text was not enough.

### Saved artifacts looked more authoritative than they were
A file like `google-auth-url.md` feels official, but OAuth URLs are ephemeral. Saving them is useful for debugging, but dangerous if anyone treats them as reusable.

## Operational recommendations for future MCP servers on Zo

### 1. Namespace secrets aggressively
Prefer app-specific names, e.g.:

- `MYAPP_GOOGLE_OAUTH_CLIENT_ID`
- `MYAPP_GOOGLE_OAUTH_CLIENT_SECRET`

Avoid raw names like:

- `GOOGLE_OAUTH_CLIENT_ID`
- `GOOGLE_OAUTH_CLIENT_SECRET`

unless you explicitly want shared ambient behavior.

### 2. Log config provenance, not secrets
At startup, log:

- which env var name supplied the client ID
- which env var name supplied redirect URI
- client ID suffix only
- final redirect URI

Do **not** log secrets.

This cuts debugging time dramatically.

### 3. Treat OAuth state as shared infrastructure
If callback and initiation can hit different processes, store state in a shared durable medium.

### 4. Never trust an old auth URL file
Mark generated auth URLs as:

- one-time
- short-lived
- invalid after use

Better: do not persist them at all unless there is a clear debugging need.

### 5. Distinguish these three things in docs
Every MCP auth setup guide on Zo should separate:

- **Google Cloud console configuration**
- **Zo service env configuration**
- **fresh runtime auth initiation**

Those are different layers and fail differently.

### 6. Prefer end-to-end verification over configuration inspection
A screenshot proving a redirect URI exists in Google Cloud is not enough. Verify:

- client ID used to generate auth URL
- redirect URI in that URL
- callback URL received
- client ID + redirect URI used during token exchange

## Recommended standard checklist for Zo-hosted OAuth MCP servers

1. Confirm the running service process is the one you think it is
2. Confirm startup logs say which env vars supplied OAuth config
3. Confirm generic provider env vars are not accidentally taking precedence
4. Generate a fresh auth URL from the live service/process
5. Check that the pending state exists in the shared state store
6. Complete the callback once only
7. If it fails, inspect callback-stage logs before retrying
8. Do not reuse old links

## Concrete guidance for future me

If an MCP OAuth flow on Zo fails:

1. **Check env precedence first**
2. **Check whether auth state exists at all**
3. **Check whether the URL being clicked is fresh**
4. Only then debug Google console settings further

This ordering is better than the instinctive “Google must be misconfigured.”

## Bottom line

The hard-earned lesson is:

> On Zo, OAuth bugs are often really **runtime wiring bugs**.

The auth provider, the hosted service environment, and the live one-time browser flow all need to line up. If you only inspect one layer, you can spend hours debugging the wrong thing.
